# craft-web

A UI skill that makes web frontend work look like it was written by a real maintainer, not generated by default AI.

## What it does

`craft-web` is a set of **constraints and a constructive visual recipe** for any web frontend task — building, fixing, refactoring, optimizing. It does **not** prescribe a fixed page structure or reply shape.

The skill has two halves:

- **Constraints** — what *not* to do. The classic AI defaults: purple gradients, glassmorphism, emoji in UI copy, oversized heroes, three-card feature grids.
- **Visual Recipe** — what *good* looks like, concretely. Specific paper colors, type scales, color palettes, spacing systems, density choices. The thing that was missing from the old "禁制清单" approach: telling the model what to ship, not just what not to ship.

Together these cover three layers:

| Layer | Covers |
|---|---|
| **Visual** | No purple gradient, no glassmorphism, no emoji in UI copy. Plus a constructive recipe: cream paper, real type systems, restrained color/spacing, 4-base scale, focus rings. |
| **Voice** | No openers, no empty apologies, no "let me think" preamble, no user-blaming on bug reports. Lead with conclusion, then reason. |
| **Code** | Business-semantic naming, no `console.log` residue, three states for data views (loading/empty/error), real a11y, no secret leakage. |

## When it triggers

The model loads this skill when the user asks anything that touches web frontend code, styling, or UX. Examples:

- "Build me a SaaS landing page"
- "Fix this UI bug — the button does nothing"
- "Optimize the interface, it looks too AI"
- "帮我做个登录页"
- "加点动效" / "改改样式"
- "数据库连不上" (when the page that displays the data is the issue)

## When it does NOT trigger

- Pure algorithm or system-design questions
- Backend-only scripts without UI
- CLI tools, data pipelines
- Research / writing tasks
- Desktop apps, mobile native, game dev

## The Visual Recipe (the core of v2)

The old version of this skill was all "don't do X". It worked for avoiding AI defaults but produced pages that were *technically not bad* and *visually forgettable*. v2 adds the constructive side — what good actually looks like, with concrete numbers.

### 1. Background and surface

Pure `#FFFFFF` reads as unfinished. Pick a paper color that has a tiny bit of warmth or cool.

| Mode | Paper | Ink | When |
|---|---|---|---|
| Light | `#FAFAF7` (warm cream) or `#F8F8F5` | `#0A0A0A` | Default for SaaS, docs, marketing |
| Light cool | `#F7F8FA` | `#0B0D12` | Dashboards, dev tools |
| Dark | `#0A0A0A` (not `#000`) | `#EDEDEA` | Dark mode default; never pure black |
| Dark rich | `#0E0F12` | `#E8E8E5` | Premium feel (Linear dark, Vercel) |

Borders: `rgba(0,0,0,0.08)` on light, `rgba(255,255,255,0.08)` on dark. Never solid 1px gray.

### 2. Type system

Two faces max (sans + optional mono). Real weights, not faux-bold via CSS.

Sans candidates (pick one family, use it everywhere):
- **Inter / Geist / Inter Tight** — default for SaaS, dashboards
- **IBM Plex Sans** — dev tools, technical
- **Söhne / Switzer** — premium (paid fonts)
- **system-ui** — only when zero asset budget

Mono candidates (when code appears in UI):
- **JetBrains Mono / Geist Mono / IBM Plex Mono**

Type scale (use this exact set unless there's a strong reason not to):

```
display    56–72px  / 600 weight / -0.02 to -0.04em tracking / line-height 1.05–1.1
h1         40–48px  / 600 / -0.015em / 1.1–1.2
h2         28–32px  / 600 / -0.01em  / 1.2–1.3
h3         20–24px  / 500–600 / -0.005em / 1.3
body       15–16px  / 400            / 1.5–1.6
caption    13–14px  / 500            / 1.4
mono       13–14px  / 400            / 1.5
```

Two weights is the rule of thumb; three is fine (400 / 500 / 600). Four is too many.

### 3. Color palette

Pattern: **1 brand + 1 accent + ink + paper + 2 grays**. That's 6. Anything more is chaos.

**Brand color**: saturated but not neon. AI defaults to purple → don't. Try ink-blue (`#1E40AF`), forest (`#166534`), or a muted coral (`#DC2626`). Saturation 60–80%, lightness 35–50%.

**Accent**: subtler than brand. Often a darker/lighter shade of brand or a near-neutral with a hint of hue.

**Grays**: one warm (`#737373` family), one cool (`#6B7280` family). Use warm for paper-on-light UIs, cool for dev tools and dashboards.

Recipe example (Linear-style):
```
brand    #5E6AD2    (a real purple-blue, not Tailwind purple-600)
accent   #4F46E5
ink      #0A0A0A
paper    #FAFAF7
gray-w   #737373
gray-c   #6B7280
```

Recipe example (Vercel-style):
```
ink      #000000
paper    #FFFFFF
accent   #0070F3
gray     #666666
```

### 4. Spacing

4-base scale, used consistently:
```
4 / 8 / 12 / 16 / 24 / 32 / 48 / 64 / 96 / 128 / 160 / 192
```

Section-to-section vertical rhythm: **96–160px**. Inside a section: **24–48px** between groups, **8–16px** between siblings. Hero to first content block: **64–96px**.

Tightness is a design choice, not a default. Linear is dense. Vercel is roomy. Pick one and apply it everywhere — mixed density reads as inconsistent.

### 5. Density

Three levels, chosen per surface:

- **Compact** (Linear, dashboard, data table): row height 32–36px, padding 8–12px, font 13–14px. Use when users scan many rows.
- **Medium** (Notion, app shell): row 40–48px, padding 12–16px, font 15px. Default for most apps.
- **Loose** (Vercel docs, marketing hero, blog): padding 24–48px, font 16–18px, line-height generous. Use for reading surfaces.

### 6. Motion

- ≤ 300ms. Default 150–200ms.
- ease-out for enter, ease-in for exit. ease-in-out only for state flips.
- No bounce. No parallax. No scroll-jacking.
- Focus ring is motion: visible on `:focus-visible`, 2px offset, brand color at 40% alpha.

### 7. Iconography

One set, one stroke width, throughout:
- **Lucide** (default, 1.5px stroke) — most things
- **Tabler** (2px stroke) — denser icons
- **Phosphor** (regular weight) — when more visual presence needed

Never mix. Never reach for emoji to substitute an icon.

### Components that signal "real"

These are the markers a real maintainer uses; AI defaults miss them:

- **Real focus ring** (not just `outline: none`)
- **Hover states** that move slightly (1px translate, subtle bg shift) — not just color flip
- **Active/pressed** state distinct from hover
- **Disabled** state with reduced opacity AND a `cursor: not-allowed`
- **Skeleton loaders** matching real layout (not generic gray rectangles)
- **Empty states** with a small illustration or icon + a real next-step CTA, not "No data"
- **Error states** with what went wrong + what to do, not a stack trace
- **Keyboard shortcuts** shown in tooltips on hover (⌘K, etc.)

## Why this version exists (the v1 → v2 lesson)

v1 of `craft-web` had a problem: pages built under it were *technically not bad* — no purple gradient, no emoji, no glassmorphism — but they were visually forgettable. The kind of page that ships as a placeholder and never gets revisited.

The cause was simple: **constraint lists tell you what to avoid, not what to ship**. A model reading "no purple gradient, no emoji, use a real type system" defaults to the most boring possible interpretation: white background, Inter, three feature cards, gray text.

v2 inverts the ratio. The **Visual Recipe is now the main act** — concrete paper colors, explicit type scales, named color recipes, density levels. The constraint list is the safety net, not the strategy.

## Calibration anchors — what to learn from each

Don't just visit these sites; learn the **specific thing** each one is good at.

- **linear.app** — density control and typographic rhythm. The compact data UI is a masterclass. Right column width matters. Button height consistency.
- **vercel.com** + **vercel.com/docs** — restrained black/white palette, generous spacing, hierarchy through size not color. The hero is the entire brand.
- **stripe.com/docs** — code samples that look like real code (not code blocks with `lorem ipsum`). Sidebar navigation as design object. Table layouts.
- **github.com** — functional UI where visual restraint serves usability. The "real maintainer" baseline: ship what's needed, nothing decorative.
- **rauchg.com** — prose-first personal site: typography, line-length, vertical rhythm. The opposite of marketing-template.
- **geist-ui.dev** — when you need a component library that already follows this recipe (Geist + Geist Mono + the spacing scale above).

Not Dribbble. Not "AI-generated landing page" inspiration boards. Those are how AI defaults get trained.

## How to evaluate

To verify a change is improving the skill (not regressing), run a producer-vs-baseline comparison:

1. Pick a real user prompt that should trigger the skill.
2. Run two parallel agents:
   - **Producer**: load `SKILL.md`, answer the prompt.
   - **Baseline**: answer the same prompt without loading the skill.
3. Compare on:
   - Did the skill produce a more structured/complete result?
   - Did it avoid unnecessary ceremony or template-matching?
   - Was the Visual Recipe visible in the producer's choices?
   - Did the producer catch "silent AI defaults" the baseline didn't notice (e.g., subtle gradients)?
   - Any regressions vs baseline (length, focus, quality)?

Stop iterating when the producer clearly beats the baseline, or when two rounds produce no meaningful gain.

In our v2 evaluation (SaaS landing page for an AI writing tool):
- Producer followed the recipe strictly (1 gradient, 0 emoji, explicit Inter/Inter Tight/JetBrains Mono)
- Baseline produced a clean-looking page but had **5 silent gradients** it never acknowledged in self-report

That's the kind of regression the recipe is designed to catch.

## Updating the skill

1. Edit `/workspace/.skills/craft-web/SKILL.md`.
2. Keep it under 500 lines. If a section grows, move it to `references/<topic>.md` and link to it.
3. Re-run the producer-vs-baseline eval after any rule change.
4. The syncer uploads the directory to OSS at session end — no manual deploy.

## Anti-patterns to avoid when modifying

- Adding `README.md` / `CHANGELOG.md` / `install.sh` / `.env` to the skill package
- Writing ALWAYS / NEVER / MUST rules without explaining why
- Stuffing the description with keywords instead of trigger phrases
- Duplicating the trigger list inside the body
- Adding scripts for things the LLM can do directly

See `skill-creator/references/anti-patterns.md` for the full list.

## What this skill is not

- **Not a design system.** Don't install shadcn just because this skill mentions it.
- **Not a Tailwind preset.** The recipe is portable to vanilla CSS, CSS Modules, Vue, whatever.
- **Not a substitute for user judgment.** When the user has a strong preference (e.g., "match our brand book"), follow that — the recipe is a default, not a law.
